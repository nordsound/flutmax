# flutmax Syntax Specification

- **Version**: 0.4.0
- **Date**: 2026-04-01
- **Parser**: Pure Rust recursive descent parser (`crates/flutmax-parser/src/parser.rs`)

## Overview

flutmax is a DSL for writing Max/MSP patch files (`.maxpat`) as text. A single `.flutmax` file corresponds to a single Max Abstraction.

## File Structure

A `.flutmax` file consists of a sequence of statements. Whitespace and newlines between statements are free-form. Comments can be inserted anywhere, as the Lexer automatically skips whitespace and comments.

```ebnf
source_file = { statement } ;
```

## Max Evaluation Model

Understanding the Max/MSP runtime environment in which the `.maxpat` generated by flutmax operates is essential for correctly reading this language specification. Max's internals consist of two independent evaluation engines.

### Dual Evaluation Model

```
+-- Control (Event) System --------------------+
|  Event-driven, sequential evaluation          |
|  bang / float / list / message                |
|  -> Node processes immediately -> propagates  |
|     output to next node                       |
|  -> Depth-first stack-based chaining          |
+-----------------------------------------------+
        <-> Boundary objects (sig~, snapshot~, etc.)
+-- Signal (MSP) System ------------------------+
|  Block-based stream processing                |
|  Audio graph is topologically sorted          |
|  -> All nodes execute every block             |
|     (64 samples)                              |
|  -> Order is statically determined by         |
|     graph structure                           |
+-----------------------------------------------+
```

### Control (Event) System

An event-driven sequential evaluation engine. There is no global evaluation order; only the parts where events arrive are evaluated.

**Factors determining evaluation order:**

1. **Inlet hot/cold attribute**: Input to a hot inlet triggers evaluation. A cold inlet only updates state
2. **Patchcord order**: Multiple connections from the same outlet have an ordering
3. **`[trigger]` object**: Explicitly controls the output order

The flutmax compiler detects Control-rate fan-outs and automatically inserts `[trigger]` to ensure the intended evaluation order.

### Signal (MSP) System

A block-based DSP processing engine. It operates on a completely different evaluation model from the Control system.

- The DSP graph is topologically sorted, and the order is statically determined
- All nodes execute every block (typically 64 samples)
- There is no dynamic event-based ordering

### Control <-> Signal Boundary

The two engines coexist within the same patch and are interconnected via boundary objects:

| Direction | Object | Behavior |
|-----------|--------|----------|
| Control -> Signal | `sig~`, `line~`, `cycle~` frequency inlet | Converts Control values to Signal |
| Signal -> Control | `snapshot~`, `edge~`, `change~` | Converts Signal to events |

The `Signal` / `Control` distinction in flutmax's Wire type is a type-level check for this dual-model boundary. `SignalFloat` inlets (such as the frequency input of `cycle~`) accept both, making them natural connection points from Control to Signal.

For a detailed analysis of Max's evaluation model and the structure needed for flutmax's intermediate representation (IR), see `docs/research/max-evaluation-model-and-ir.md`.

## Significance of Statement Order

The source code order in flutmax is used for compile-time dependency resolution and the layout order of the generated Max patch. This order primarily affects placement for readability and does not directly dictate Max's runtime evaluation order.

Specifically:

- **Dependency resolution**: Each wire can only reference wires, ports, and feedbacks declared before it
- **Layout**: Statement order is reflected in the Y-coordinate (top-to-bottom placement) of the generated boxes
- **Not execution order**: Runtime evaluation order follows the semantics of each Max object, inlet/outlet hot/cold attributes, patchlines, and patchcord order as needed

The flutmax compiler automatically inserts `[trigger]` for Control-rate fan-outs, but this leverages Max's evaluation order rules to ensure the intended order -- the statement order in flutmax does not directly become the execution order.

### Implicit Trigger Insertion Rules (ADR-006)

When the same wire is passed as an argument to multiple `object_call`s, the compiler automatically inserts `[trigger]`. **Earlier in the source is executed first**:

```flutmax
wire src = button();
wire a = print(src);   // Executed 1st -> trigger outlet 0 (rightmost)
wire b = counter(src); // Executed 2nd -> trigger outlet 1 (leftmost)
```

Trigger is not inserted for Signal-rate fan-outs (DSP graph order is irrelevant).

### Max Fan-out Execution Order (Reference)

The execution order when Max sends from the same outlet to multiple destinations:

1. **1st priority: X-coordinate** -- Further right (larger X) executes first (Right to Left)
2. **2nd priority: Y-coordinate** -- Further down (larger Y) executes first (Bottom to Top)

The flutmax compiler controls the statement order (top-to-bottom) via trigger outlet mapping, so it does not depend on Max's coordinate-based ordering. The decompiler analyzes the coordinate ordering of the original patch and reflects it in the flutmax statement order.

### Explicit Trigger

This can be handled with the existing destructuring syntax:

```flutmax
wire (bang_out, float_out) = t(src);
wire a = process(bang_out);
wire b = calc(float_out);
```

## Statements

Ten types of statements are defined:

```ebnf
statement = port_declaration
          | wire_declaration
          | destructuring_wire
          | out_assignment
          | direct_connection
          | feedback_declaration
          | feedback_assignment
          | state_declaration
          | state_assignment
          | msg_declaration ;
```

## 1. Port Declaration

Declares the input/output ports of an Abstraction. Corresponds to Max's `[inlet]`/`[outlet]`.

### Syntax

**Implicit port index** syntax was added in v0.4.0 (E44). Two forms are supported:

```ebnf
port_declaration = direction ( explicit_port | implicit_port ) ;
explicit_port    = INTEGER "(" IDENTIFIER ")" ":" type_name ";" ;
implicit_port    = IDENTIFIER ":" type_name ";" ;
direction        = "in" | "out" ;
```

**Explicit port**: The index is specified numerically. This is the legacy syntax (backward-compatible).
**Implicit port**: The index is omitted and automatically assigned in declaration order (sequential from 0). This is the default output of the decompiler.

Parser determination: If the token immediately after `in`/`out` is a number, it uses explicit mode; if it is an identifier, it uses implicit mode.

### Type Names (Port type)

| Type Name | Meaning in Max |
|-----------|---------------|
| `signal` | Audio rate (MSP) |
| `float` | Floating-point number |
| `int` | Integer |
| `bang` | Trigger |
| `list` | List |
| `symbol` | Symbol |

### Examples

```flutmax
// Implicit port index (v0.4.0 recommended, index determined by declaration order)
in freq: float;          // -> index 0
in cutoff: float;        // -> index 1
out audio: signal;       // -> index 0
out monitor: float;      // -> index 1

// Explicit port index (backward-compatible, use when special reordering is needed)
in 0 (freq): float;
in 1 (cutoff): float;
out 0 (audio): signal;
out 1 (monitor): float;
```

### Safety of Implicit Port Indices

The decompiler sorts Max patch inlets/outlets by X-coordinate order for output, so declaration order = index order is always guaranteed. When writing manually, declaration order = port order is also natural. Use explicit indices only for special cases (reordering).

### Generated Max Objects

- `in name: signal;` / `in N (name): signal;` -> `maxclass: "inlet"`, `outlettype: ["signal"]`
- `in name: float;` / `in N (name): float;` -> `maxclass: "inlet"`, `outlettype: [""]`
- `out name: signal;` / `out N (name): signal;` -> `maxclass: "outlet"`
- `out name: float;` / `out N (name): float;` -> `maxclass: "outlet"`

## 2. Wire Declaration

Defines intermediate variables (wires) and generates Max objects and their connections.

### Syntax

```ebnf
wire_declaration = "wire" IDENTIFIER "=" expression [ ".attr(" attr_list ")" ] ";" ;
attr_list        = attr_pair { "," attr_pair } ;
attr_pair        = IDENTIFIER ":" attr_value ;
attr_value       = number | string | IDENTIFIER ;
```

### Examples

```flutmax
wire osc = cycle~(440);
wire amp = mul~(osc, 0.5);
wire gain = mul(freq, 2);
wire knob = flonum(freq).attr(minimum: 20., maximum: 20000.);
```

### Semantics

- `wire IDENTIFIER = expression;` generates:
  1. A Max object (a box corresponding to the `expression`)
  2. Connections (patchlines) from the referenced sources of the arguments
- The wire name can be referenced in subsequent statements
- Redefining a wire with the same name is not allowed (**E003 error**)

---

## 3. Destructuring Wire

Decomposes a list into multiple wires. Corresponds to Max's `[unpack]`.

### Syntax

```ebnf
destructuring_wire = "wire" "(" identifier_list ")" "=" expression ";" ;
identifier_list    = IDENTIFIER "," IDENTIFIER { "," IDENTIFIER } ;
```

### Examples

```flutmax
wire (x, y, z) = unpack(coords);
wire (left, right) = unpack(stereo_signal);
```

### Generated Max Object

- `[unpack f f f]` (with outlets corresponding to the number of elements)
- Each wire is connected to the corresponding outlet

### Type Rules

- Each wire generated by destructuring has a Wire type of **Control**
- If the right-hand side is Signal, it is an error -- Max's `[unpack]` is a Control-rate object
- The fine-grained types of each element (float / int / symbol) are not distinguished; all are treated as Control (the Wire type layer of the two-tier type system)
- If the number of outlets on the right-hand side can be statically determined and does not match the number of IDENTIFIERs on the left-hand side, it is an **E006 error**. If the outlet count cannot be statically determined (e.g., unknown object), the check is deferred

---

## 4. Output Assignment

Connects a wire to an output port.

### Syntax

```ebnf
out_assignment = "out" "[" INTEGER "]" "=" expression ";" ;
```

### Examples

```flutmax
out[0] = amp;
out[1] = monitor_val;
```

### Semantics

- `out[N]` corresponds to the port declared with `out N (name): type;`
- If no corresponding `out` declaration exists, it is an **E004 error**
- If the Wire type and Port type are incompatible, it is an **E005 error**

---

## 5. Direct Connection

Directly connects to a port of an object instance generated by a wire.

### Syntax

```ebnf
direct_connection   = input_port_access "=" expression ";" ;
input_port_access   = IDENTIFIER ".in[" INTEGER "]" ;
```

`input_port_access` can **only be used as an assignment target (lvalue)** and does not appear within expressions.

Use `output_port_access` to extract values (see the Expressions section).

### Semantics

An object instance generated by `wire name = object_call(...)` can be referenced by `name`.

- `name.in[N]` is a **connection target** (lvalue) to inlet N
- `name.out[N]` is a **value extraction** (rvalue, used as an expression) from outlet N
- `name` must be a wire name
- Referencing a non-existent inlet/outlet results in an **E007 error**

### Duplicate Connections to the Same Inlet

Argument connections (`wire x = obj(a, b)`) and direct connections (`obj.in[0] = c;`) overlapping on the same inlet are **permitted** (since Max allows multiple connections to the same inlet). However:

- The compiler issues a **warning (W001)** for duplicate connections
- Patchcord order is determined by statement order (top-to-bottom)
- Automatic trigger insertion applies for Control-rate duplicate connections

### Examples

```flutmax
wire node_a = cycle~(440);
wire node_b = biquad~(node_a, 1000);

node_a.in[0] = trigger;   // Connect to node_a's inlet 0 (lvalue)
node_b.in[1] = cutoff;    // Connect to node_b's inlet 1 (lvalue)
```

## 6. Feedback Declaration

Forward declaration for feedback loops. Used in combination with `tapin~`/`tapout~`. This is a **Signal-only syntax**.

### Syntax

```ebnf
feedback_declaration = "feedback" IDENTIFIER ":" type_name ";" ;
feedback_assignment  = "feedback" IDENTIFIER "=" expression ";" ;
```

Feedback is **fixed to Signal only** (ADR-002). While the grammar accepts `type_name`, specifying anything other than `signal` results in an **E010 error**. Control-rate state persistence will be introduced separately as the `state` syntax in v0.5, and feedback will not be extended to Control. Functionality equivalent to `[send]/[receive]` or `[send~]/[receive~]` will be considered separately as future `channel`/`bus` syntax.

### Constraints

- Assignment to each feedback name is allowed **only once**. Multiple `feedback name = ...;` statements result in an **E013 error**
- If a `feedback` declaration has no corresponding assignment, it is an **E011 error**
- Assignment to an undeclared feedback name is an **E012 error**

### Examples

```flutmax
feedback fb: signal;
wire delayed = tapout~(fb, 500);
wire mixed = add~(input, mul~(delayed, 0.3));
feedback fb = tapin~(mixed, 1000);
```

### Semantics

- `feedback name: signal;` forward-declares a feedback variable
- `tapout~(fb, delay)` reads the feedback variable (with delay)
- `feedback name = tapin~(expr, buffer);` writes to the feedback variable
- In the generated .maxpat, `[tapin~]` and `[tapout~]` are cyclically connected
- Feedback edges are excluded from topological sorting

---

## 7. State Declaration

Holds Control-rate local state. Unlike Signal's `feedback`, it preserves values across events. **Control only** (ADR-002).

### Syntax

```ebnf
state_declaration = "state" IDENTIFIER ":" control_type "=" expression ";" ;
state_assignment  = "state" IDENTIFIER "=" expression ";" ;
control_type      = "float" | "int" | "bang" | "list" | "symbol" ;
```

### Examples

```flutmax
state counter: int = 0;
wire next = add(counter, 1);
state counter = next;
out[0] = next;
```

### Semantics

- `state name: type = init;` declares local state with an initial value. Corresponds to Max's `[int]`/`[float]`
- `name` in an expression reads the current state value
- `state name = expr;` updates the state within an event chain
- Signal cannot be used (**E017 error**)
- Assignment to the same state is limited to once (**E019 error**)
- State names belong to the value namespace

---

## 8. Message Box Declaration

Declares a Max message box (`maxclass: "message"`).

### Syntax

```ebnf
msg_declaration = "msg" IDENTIFIER "=" STRING ";" ;
```

### Examples

```flutmax
msg click = "bang";
msg format = "set $1 $2";
msg note = "60 100";
```

### Semantics

- `msg name = "content";` generates a Max `[message]` box
- `name` belongs to the value namespace and can be referenced like a wire
- Wire type is `Control(Symbol)`
- inlet 0 (hot): outputs the message on bang. inlet 1 (cold): `set` message
- outlet 0: outputs the message content

## Expressions

### Syntax

```ebnf
expression = object_call
           | output_port_access
           | tuple_expression
           | identifier
           | number
           | string ;

output_port_access = IDENTIFIER ".out[" INTEGER "]" ;
```

`output_port_access` can **only be used as a value extraction (rvalue)**. Use `input_port_access` for assignment targets (lvalue) (see the Direct Connection section).

### Identifier Reference (identifier)

An `identifier` in an expression refers to one of the following:

| Reference Target | Example | Description |
|------------------|---------|-------------|
| **wire name** | `osc`, `amp` | An intermediate variable declared with `wire IDENTIFIER = ...;` |
| **port name** | `freq`, `cutoff` | An input port name declared with `in N (IDENTIFIER): type;` |
| **feedback name** | `fb` | A feedback variable declared with `feedback IDENTIFIER: type;` |
| **destructuring name** | `x`, `y` | A destructuring variable declared with `wire (IDENTIFIER, ...) = ...;` |

**There is a single value namespace.** Port names, wire names, feedback names, and destructuring names all belong to the same namespace, and **duplicates are prohibited** (E003 error). Shadowing is not allowed.

Abstraction names (other `.flutmax` files) do not belong to the value namespace. Abstractions are used only in the `object_name` position of an object call (`object_call`).

```flutmax
in 0 (freq): float;
wire osc = cycle~(freq);       // freq is a reference to the port name
wire amp = mul~(osc, 0.5);     // osc is a reference to the wire name
wire out_sig = oscillator(amp); // oscillator is an Abstraction (object_call)

// wire freq = ...;             // E003: 'freq' is already defined as a port name
```

Referencing a non-existent name results in an **E002 error**.

#### Bare Reference and `.out[N]`

A bare wire name (`name`) implicitly means `name.out[0]`. This is always valid regardless of the number of outlets. Use `.out[N]` only when accessing outlet 1 or later of a multi-outlet node.

```flutmax
wire osc = cycle~(440);          // bare reference = osc.out[0]
wire amp = mul~(osc, 0.5);       // OK

wire polar = cartopol~(x, y);    // cartopol~ has 2 outlets
wire mag = polar;                 // OK: polar.out[0] (magnitude)
wire phase = polar.out[1];        // .out[1] is needed for outlet 1 (phase)
```

### Object Call

```ebnf
object_call   = object_name "(" [ argument_list ] ")" ;
object_name   = ( IDENTIFIER | OPERATOR ) { "." segment } [ "~" ] ;
segment       = IDENTIFIER | OPERATOR | KEYWORD | NUMBER_IDENT ;
argument_list = expression { "," expression } ;
```

#### Operator Name Objects (v0.4.0, E42)

In gen~ / RNBO internal patchers, operator characters such as `?`, `*`, `+`, `-`, `/`, `%`, `==`, `!=`, `<=`, `>=`, `&&`, `||` are used as object names. The v0.4.0 parser recognizes these as `Operator` tokens and permits them at the beginning of object names and within `.`-delimited segments.

```flutmax
// Ternary operator object in gen~
wire cond = ?(input, val_true, val_false);

// Operator objects in gen~
wire sum = +(a, b);
wire product = *(x, y);
wire compare = ==(left, right);

// mc namespace operators: mc.+~, mc.*~
wire mc_sum = mc.+~(sig_a, sig_b);
```

Operator characters: `*`, `+`, `/`, `%`, `!`, `<`, `>`, `=`, `&`, `|`, `^`, `?`

In regular Max patchers, it is also possible to use operator objects (`*`, `+`, etc.) directly, but the use of named aliases (`mul`, `add`, etc.) as described in ADR-001 is recommended. Within gen~, operator names are native and should be used as-is.

#### Contextual Use of Keywords (v0.4.0, E43)

flutmax keywords (`in`, `out`, `int`, `float`, `bang`, `list`, `symbol`, `signal`, `state`, `msg`, `feedback`, `wire`) are treated as **contextual keywords**. In expression contexts, they can be used as object names or arguments:

```flutmax
// Send an int message to Max's thispatcher object
wire tp = thispatcher(int);

// Use float keyword as arguments to pack
wire packed = pack(float, float, float);

// gen~ in / out objects
wire gen_input = in(1);
wire gen_output = out(result, 1);
```

They function as keywords only at statement-initial positions (`wire`, `in`, `out`, `msg`, `feedback`, `state`).

### Object Name Resolution Order

The `object_name` of an `object_call` is resolved in the following priority order:

| Priority | Target | Example |
|----------|--------|---------|
| 1 | flutmax arithmetic aliases | `mul` -> `*`, `add~` -> `+~` |
| 2 | Abstraction in the same compilation unit | `oscillator` -> `oscillator.flutmax` |
| 3 | Max object (as-is) | `cycle~`, `biquad~`, `pack` |

When resolved as an Abstraction, `numinlets`/`numoutlets` are determined from the referenced file's `in`/`out` declarations. Checks performed:

- If the number of arguments does not match the Abstraction's inlet count, it is an **E009 error**
- The Wire type of each argument and the Port type of the corresponding `in` declaration in the referenced Abstraction are checked for compatibility (e.g., passing a Signal argument to `in N: float` results in an **E001 error**)

### Arithmetic Aliases (ADR-001)

Named aliases are used instead of bare symbols (`*`, `+`):

| flutmax | Max Object | Description |
|---------|-----------|-------------|
| `add` / `add~` | `+` / `+~` | Addition |
| `sub` / `sub~` | `-` / `-~` | Subtraction |
| `mul` / `mul~` | `*` / `*~` | Multiplication |
| `dvd` / `dvd~` | `/` / `/~` | Division |
| `mod` / `mod~` | `%` / `%~` | Modulo |

Codegen automatically converts these to Max object names.

### Tuple Expression

Wrapping two or more elements in parentheses creates a tuple (Max's `[pack]`).

```ebnf
tuple_expression = "(" expression "," expression { "," expression } ")" ;
```

```flutmax
wire packed = (x, y, z);  // -> [pack f f f]
```

#### Type Constraints for Tuple Expressions

- **Each element must be Control. Signal is not permitted** (**E008 error**)
- The Wire type of the entire tuple expression is `Control`
- Mixing different Control subtypes (`float`, `int`, `string`) is allowed -- Max's `[pack]` can produce heterogeneous lists
- In the v0.2.0 specification, tuple expressions are always converted to `[pack f f ...]` (treating all elements as float). Generating typed pack such as `[pack i f s]` based on element types is under consideration for a future version

```flutmax
// OK: Control elements only
wire rgb = (255, 128, 0);
wire mixed = (1, 0.5, "label");

// ERROR: Tuples containing Signal are prohibited
wire osc = cycle~(440);
wire bad = (osc, 100);  // E008: signal in tuple
```

## Literals

At the lexical level, these are uniformly tokenized as `NumberLit`. Semantically, they are represented in the AST as `LitValue::Int(i64)` or `LitValue::Float(f64)`.

### Number Literals

**Scientific notation** (exponential notation) and **trailing dot** are supported in v0.4.0 (E40):

```ebnf
number     = [ "-" ] DIGIT { DIGIT } [ "." { DIGIT } ] [ exponent ] ;
exponent   = ( "e" | "E" ) [ "+" | "-" ] DIGIT { DIGIT } ;
```

#### Integers

Examples: `440`, `0`, `100`, `-7`

#### Floating-Point Numbers

Examples: `0.5`, `3.14`, `1.0`, `100.` (trailing dot)

#### Scientific Notation

Examples: `1e-6`, `3.14e5`, `2.5E+3`, `1e10`

Scientific notation is particularly useful for precise numeric specification in `.attr()`:

```flutmax
wire knob = flonum(freq).attr(minimum: 1e-6, maximum: 2e4);
wire param = param("decay").attr(minimum: 0.001, maximum: 3.14e5);
```

If a decimal point or exponent part is present, the value is treated as floating-point. `440` is an integer; `440.0`, `440.`, `4.4e2` are floating-point.

### Strings

```ebnf
string = '"' { CHAR | ESCAPE } '"' ;
ESCAPE = "\\" ANY ;
```

Examples: `"hello"`, `"debug: value is"`

## Identifiers

### Regular Identifiers (IDENTIFIER)

Extended in v0.3.0 (ADR-003). Hyphens are permitted:

```ebnf
SIMPLE_IDENT = LETTER { LETTER | DIGIT | "_" | "-" } ;
LETTER       = "a"-"z" | "A"-"Z" | "_" ;
```

Examples: `freq`, `cutoff`, `osc1`, `my_wire`, `drunk-walk`

The v0.4.0 Lexer outputs each segment as an independent token, with dot `.` also being an independent token. The Parser assembles dot-delimited names (`jit.gl.videoplane`, `live.dial`, `M4L.api.getcount`, etc.) as object names. This supports Max namespaces (Jitter `jit.*`, Max for Live `live.*`, `M4L.*`, `zl.*`, etc.).

Collision with the `.in[N]` / `.out[N]` port access syntax is resolved by the parser via lookahead (`.in[` / `.out[` takes priority as port access; other dot-delimited tokens continue as part of the object name).

### Tilde-Suffixed Identifiers (TILDE_IDENTIFIER)

```ebnf
TILDE_IDENTIFIER = object_name "~" ;  (* no whitespace allowed *)
```

Examples: `cycle~`, `mul~`, `biquad~`, `mc.+~`

No whitespace is permitted between the tilde and the identifier (the parser checks adjacency). The v0.4.0 Lexer outputs the tilde as an independent `Tilde` token, and the Parser checks adjacency with the preceding token to combine them into a name.

---

## Comments

Only line comments are supported. Comments are automatically skipped by the Lexer along with whitespace, so they can be inserted at any position between statements or expressions.

```ebnf
comment = "//" { ANY except NEWLINE } ;
```

```flutmax
// This is a comment
wire osc = cycle~(440);  // Inline comment

// Japanese comments are also supported
// Unicode characters can be used freely
```

## Type System

The flutmax type system is intentionally designed with a **two-tier structure**.

### Design Intent of the Two-Tier Structure

| Tier | Name | Purpose | Granularity |
|------|------|---------|-------------|
| **Port type** | Port Type | Abstraction boundary (`in`/`out` declarations) | Fine: 6 types |
| **Wire type** | Wire Type | Internal connection type checking | Coarse: 3 types |

**Port type** (`signal`, `float`, `int`, `bang`, `list`, `symbol`) precisely describes an Abstraction's interface. This allows the referencing side to know exactly "what should be connected to this port."

**Wire type** (`Signal`, `Control`, `Unknown`) is a coarse classification used for internal wiring type checks. It corresponds to Max/MSP's fundamental dichotomy (Signal = audio rate vs Control = message rate).

```
Port type -> Wire type mapping:

  signal                  ->  Signal
  float / int / bang      ->  Control
  list / symbol           ->  Control
  (unknown object output) ->  Unknown
```

`float`, `int`, `bang`, `list`, `symbol` are all unified to `Control` in internal wiring. This reflects the reality of Max/MSP -- in Max's message-rate communication, `float` and `int` are implicitly converted, and `bang` can be sent to any Control inlet.

There is a possibility of introducing finer types at the wire level in the future (e.g., distinguishing `Bang` / `Int` / `Float` / `List`), but at the current stage, the Signal vs Control binary provides sufficient error detection, and the coarse classification is adopted to avoid complexity.

### Wire Type Inference

| Expression | Inferred Wire type |
|------------|-------------------|
| Signal object call (`~` suffix) | Signal |
| Control object call | Control |
| Reference to Signal wire | Signal |
| Reference to Control wire | Control |
| Literal value (integer, float, string) | Control |
| Tuple expression | Control |
| Each element of destructuring | Control |
| Unknown object (not registered in objdb) | Unknown |

### Type Compatibility Rules

Internal wiring type checks are performed at the Wire type level:

| Source wire type | Dest inlet | Result | Reason |
|------------------|-----------|--------|--------|
| Signal | Signal / SignalFloat | OK | Connection between MSP objects |
| Signal | Control (non-UI) | **E001 error** | Signal cannot be connected to a pure Control object's inlet |
| Signal | Control (UI) | **W002 warning** | Max allows Signal -> UI connections via implicit conversion. Signal connections to UI objects such as `number`, `flonum`, `multislider`, `slider`, `dial`, `toggle`, `button` are legal, but explicit use of `snapshot~` is recommended |
| Control | Signal / SignalFloat | OK | float messages are accepted by SignalFloat inlets |
| Control | Control / Any | OK | Normal message connection |
| Unknown | any | OK | Unknown objects skip checking |

Max performs implicit Signal-to-Control conversion when connecting Signal to UI objects. flutmax permits this but outputs a warning (W002) recommending explicit conversion (using `snapshot~`). For strict type safety, this can be promoted to E001 in strict mode.

At Abstraction boundaries (`out[N] = expr`), checking is performed at the Port type level:

| Wire type | Out port type | Result |
|-----------|--------------|--------|
| Signal | `signal` | OK |
| Signal | `float` etc. | **E005 error** |
| Control | `float` / `int` / `bang` / `list` / `symbol` | OK |
| Control | `signal` | **E005 error** |

## Abstraction Integration

### Principle

A single `.flutmax` file corresponds to a single Max Abstraction.

### Object Name Resolution

The `object_name` of an `object_call` is resolved in the following priority order:

| Priority | Target | Example | Resolution Result |
|----------|--------|---------|-------------------|
| 1 | flutmax arithmetic aliases | `mul`, `add~` | Converted to Max object name (`*`, `+~`) |
| 2 | Abstraction in the same compilation unit | `oscillator` | Obtains inlet/outlet information from `AbstractionRegistry` |
| 3 | Max object (as-is) | `cycle~`, `biquad~` | Output directly to `.maxpat` |

Names that do not match either priority 1 or 2 are output as-is to `.maxpat` as Max object names. Max resolves the object at runtime.

### Compilation

Batch compilation in directory mode:

```bash
flutmax compile src/ -o dist/
```

1. Parse all `.flutmax` files and build `AbstractionRegistry` from `in`/`out` declarations
2. Compile each file with the registry reference
3. Output `.maxpat` files in the same directory

## Global Communication: send / receive (ADR-006)

A Max mechanism for exchanging data without using patchline connections. Treated as regular object calls:

```flutmax
// Control rate
wire s = send(data, "my-channel");
wire r = receive("my-channel");

// Signal rate (1 vector delay)
wire s = send~(audio, "my-bus");
wire r = receive~("my-bus");

// Global shared variable
wire v = value("shared-data");

// Patcher-local variable
wire pv = pv("local-data");
```

`send`/`receive` names are shared across all of Max (global scope). They can be made instance-local with the `#0` prefix (see ADR-005).

flutmax does not verify `send`/`receive` name matching at compile time (this is delegated to the Max runtime).

### Regarding `#0` Verification

`#0` is a Max runtime feature that is replaced with a unique integer for each poly~ voice, and it is fundamentally impossible to know the actual value at compile time. Therefore, typos in names like `#0-mydata` cannot be discovered until runtime.

This is the same constraint as Max itself, and is accepted within flutmax's current scope. The following can be considered as future lints:

- **W005**: Warning when a `receive("name")` corresponding to `send("name")` does not exist in the same file
- **W006**: Warning when one half of a `#0-` prefixed send/receive pair is missing (pattern matching)

## External Code References: js / gen~ (ADR-006)

Max's `[js]` and `[gen~]` contain code in another language internally. In flutmax, these are treated as external file references.

**Principle of reference style**: `.flutmax` files compiled by flutmax are referenced by **identifiers**, and external files not managed by flutmax are referenced by **string arguments**.

| Reference Target | Syntax | Managed by flutmax |
|------------------|--------|:-----------------:|
| Abstraction | `oscillator(440)` | Yes (identifier) |
| poly~ voice | `poly~(myvoice, 4)` | Yes (identifier) |
| pfft~ process | `pfft~(myprocess, 1024)` | Yes (identifier) |
| gen~ patcher | `gen~(my-gen-patch)` | Yes (identifier) |
| rnbo~ patcher | `rnbo~(my-rnbo-patch)` | Yes (identifier) |
| js script | `js("script.js")` | No (string) |
| node.script | `node.script("server.mjs")` | No (string) |
| v8.codebox | `v8.codebox("code.js")` | No (string) |
| gen~ codebox | `codebox("file.genexpr")` | No (string) |

```flutmax
wire logic = js("my-script.js");
wire n4m = node.script("server.mjs");
wire dsp = gen~(my-gen-patch);
wire rnbo = rnbo~(my-rnbo-patch);
```

## gen~ / RNBO / codebox Support (E33, E37, E39, E42)

Compilation and decompilation of gen~ patchers, RNBO patchers, and codebox are supported in v0.4.0.

### gen~ Patcher

gen~ is Max's DSP code generation environment. Internal patchers have `classnamespace: "dsp.gen"`, and all objects operate at signal rate.

#### Calling from Parent Patch

```flutmax
wire gen_dsp = gen~(my-gen-patch);
wire gen_fx = gen~(my-gen-effect).attr(gen_patcher_name: "my-gen-effect");
```

gen~ internal patchers are represented as separate `.flutmax` files.

#### Characteristics of gen~ Internal Patchers

- I/O: `in N` / `out N` (as Max newobj text, not regular `inlet`/`outlet`)
- All objects operate at signal rate (no `~` suffix)
- Operator objects: `*`, `+`, `-`, `/`, `?`, etc. are used as-is
- gen~-specific objects: `param`, `history`, `delay`, `buffer`, `data`, etc.

```flutmax
// my-gen-patch.flutmax -- gen~ internal patcher
in gen_in_1: signal;
in gen_in_2: signal;
out gen_out_1: signal;

wire phasor = phasor(gen_in_1, gen_in_2);
wire noise = noise();
wire param = param("multiplier");
wire product = *(phasor, param);
wire cond = ?(product, noise, gen_in_1);

out[0] = cond;
```

#### gen~ Mode Type Checking (E42)

Since all objects operate at signal rate within gen~, the Signal/Control distinction is unnecessary. The compiler skips E001 (Signal -> Control mismatch) and E005 (output type mismatch) type checks in gen~ mode.

### RNBO Patcher

RNBO is a compiler environment that enables exporting Max patches to C++/JavaScript/Raspberry Pi, etc. Internal patchers have `classnamespace: "rnbo"`.

#### Calling from Parent Patch

```flutmax
wire synth = rnbo~(my-rnbo-patch).attr(polyphony: 4);
```

#### Characteristics of RNBO Internal Patchers

- I/O: `inport` / `outport` (not regular `inlet`/`outlet`)
- Signal I/O: `in~ N` / `out~ N`
- RNBO-specific fields: `rnbo_serial`, `rnbo_uniqueid` (retained as `.attr()`)
- RNBO-specific objects: `param`, `buffer~`, `midiin`, `midiout`

```flutmax
// my-rnbo-patch.flutmax -- RNBO patcher
in frequency: float;
out audio: signal;

wire osc = cycle~(frequency);
out[0] = osc;
```

### codebox

codebox is an object that embeds text code within a Max patch. In flutmax, code content is extracted as an external file and referenced by file path.

| Type | maxclass | Code Language | External File Extension |
|------|----------|--------------|------------------------|
| gen~ codebox | `codebox` | GenExpr | `.genexpr` |
| jit.gen codebox | `codebox` | GenExpr | `.genexpr` |
| v8.codebox | `v8.codebox` | JavaScript (V8) | `.js` |
| RNBO codebox | `codebox` | RNBO GenExpr | `.genexpr` |

```flutmax
// Using codebox within gen~
wire cb = codebox("my-patch_gen_obj7.genexpr");

// Using v8.codebox at the top level
wire v8 = v8.codebox("my-patch_codebox_obj4.js");
```

During decompilation, the contents of the `code` field are extracted as an external file, and during compilation, the external file is read and embedded into the `code` field. This allows codebox code to be managed individually in Git.

## Tolerance for Incomplete Patches (ADR-006)

Compilation of incomplete patches during development is permitted:

- **Unused wires**: Compilation succeeds. **W003 warning**. Retained in the Max patch (not deleted)
- **Unconnected inlets**: Permitted. Operates with Max's default values
- **Unconnected out declarations**: **W004 warning**

```flutmax
wire unused = cycle~(440);  // W003: wire 'unused' is never referenced
                             // [cycle~ 440] is generated in the Max patch
```

## poly~ / pfft~ (ADR-005)

### poly~ -- Polyphonic Processing

`poly~` is an object that manages multiple voice instances. Internal patchers are represented as separate `.flutmax` files.

#### Calling from Parent Patch

```flutmax
wire voices = poly~(myvoice, 4);
wire synth = poly~(fm_voice, 8).attr(steal: 1, target: 0);
```

- 1st argument: voice patcher name (`.flutmax` file in the same directory)
- 2nd argument: number of voices
- `.attr()` specifies poly~ options (steal, target, etc.)

#### Voice Patcher I/O

Within a voice patcher, channel numbers are passed as arguments to `in~` / `out~` / `in` / `out`:

```flutmax
// myvoice.flutmax -- poly~ voice patcher
wire sig_in = in~(1);           // Signal input channel 1
wire msg_in = in(2);            // Message input channel 2
wire processed = cycle~(sig_in);
wire sig_out = out~(processed, 1);  // Signal output channel 1
wire msg_out = out(status, 2);      // Message output channel 2
```

Channel numbers correspond to the inlet/outlet order of the poly~ parent object. Signal outlets are placed to the left of Message outlets.

#### Voice State Control (`thispoly~`)

```flutmax
wire poly_ctl = thispoly~();
// outlet 0: voice number
// inlet 0: mute message
```

`thispoly~` is treated as a regular object call.

#### `#0` Local Namespace

`#0` is a Max runtime feature that is replaced with a unique integer for each poly~ voice. In flutmax, it is retained as a string literal and is not substituted at compile time:

```flutmax
wire s = send(data).attr(name: "#0-mydata");
wire r = receive().attr(name: "#0-mydata");
```

### pfft~ -- Spectral Processing

`pfft~` is an object for FFT-based spectral processing. Internal patchers operate in the spectral domain.

#### Calling from Parent Patch

```flutmax
wire spectral = pfft~(myprocess, 1024, 4);
```

- 1st argument: FFT processing patcher name
- 2nd argument: FFT size
- 3rd argument: overlap

#### FFT Processing Patcher I/O

```flutmax
// myprocess.flutmax -- pfft~ spectral processing
wire fft_in = fftin~(1);       // outlet 0: real, outlet 1: imaginary
wire info = fftinfo~();         // FFT size and other parameters

// Spectral processing
wire polar = cartopol~(fft_in.out[0], fft_in.out[1]);
wire scaled = mul~(polar.out[0], 0.5);
wire result = poltocar~(scaled, polar.out[1]);

// FFT output
wire fft_out = fftout~(result.out[0], 1);
fft_out.in[1] = result.out[1];
```

pfft~-specific objects:

| Object | Description |
|--------|-------------|
| `fftin~ N` | Spectral input. outlet 0 = real, outlet 1 = imaginary |
| `fftout~ N` | Spectral output. inlet 0 = real, inlet 1 = imaginary |
| `fftinfo~` | FFT size, hop size, sigvs, fullspectrum flag |

## `.uiflutmax` Sidecar File (E44)

Introduced in v0.4.0. A sidecar file format for separating decorative attributes and object positions from logic.

### Purpose

Separates decorative attributes (`background`, `bgcolor`, `fontsize`, etc.) and object positions (`patching_rect`) that were included in `.flutmax` files from the logic, making code review via Git diff easier.

```
patch.flutmax          <- Logic + functional attributes (primary target for Git management)
patch.uiflutmax        <- Position + decorative attributes (merged at compile time if present)
patch_codebox_obj4.js  <- codebox code (if present)
```

### File Format

`.uiflutmax` is in JSON format, storing object positions and decorative attributes with wire names as keys:

```json
{
  "_patcher": {
    "rect": [100, 100, 640, 480],
    "gridsize": [15, 15]
  },
  "envelope": {
    "rect": [200, 300, 200, 100],
    "background": 0,
    "bordercolor": "0.0 0.0 0.0 1.0",
    "linecolor": "0.333 0.333 0.333 1.0"
  },
  "carrier": {
    "rect": [100, 150, 80, 22]
  }
}
```

### Key Structure

| Key | Purpose |
|-----|---------|
| `_patcher` | Patcher-level settings (window size, grid, etc.) |
| wire name | Position and decorative attributes for each object |

### Attribute Classification

| Category | Examples | Stored In |
|----------|---------|-----------|
| **Logic** | `@min`, `@max`, `domain`, `range`, `addpoints` | `.flutmax` `.attr()` |
| **Decorative** | `background`, `bgcolor`, `fontsize`, `textcolor`, `bordercolor` | `.uiflutmax` |
| **Position** | `patching_rect` [x, y, w, h] | `.uiflutmax` |
| **Patcher settings** | Window size, grid settings | `.uiflutmax` `_patcher` |

### Behavior

- **No `.uiflutmax`**: Codegen auto-layout + default decoration (fallback behavior)
- **With `.uiflutmax`**: Restores original patch coordinates and decoration (full reproduction)
- **Git diff**: Logic changes affect only `.flutmax`, and visual changes are isolated to `.uiflutmax`

## Wire Naming Convention (E44)

Wire naming scheme for the decompiler was overhauled in v0.4.0.

### Object Name-Based Naming

Wire names generated by the decompiler are derived from Max object names (replacing the previous `w_N` sequential numbering scheme):

| Priority | Source | Example | Coverage |
|----------|--------|---------|----------|
| 1 | Has varname | `envelope`, `FreqMode_3_` | ~4% |
| 2 | Object name | `cycle~` -> `cycle`, `biquad~` -> `biquad` | ~96% |
| 3 | Sequential number on name collision | `cycle`, `cycle_2`, `cycle_3` | --- |

#### Derivation Rules

1. If the Max object has a varname, use it
2. Otherwise, remove the `~` suffix from the object name
3. Operator names are converted to named aliases: `*~` -> `mul`, `+~` -> `add`, `-~` -> `sub`, `/~` -> `dvd`, `%~` -> `mod`
4. If the same name already exists, append `_2`, `_3`, ...

```flutmax
// Before (v0.3.0)                 // After (v0.4.0)
wire w_1 = cycle~(440);            wire cycle = cycle~(440);
wire w_2 = biquad~(w_1);           wire biquad = biquad~(cycle);
wire w_3 = gain~(w_2);             wire gain = gain~(biquad);
wire w_4 = cycle~(880);            wire cycle_2 = cycle~(880);
wire w_5 = snapshot~(w_3);         wire snapshot = snapshot~(gain);
```

### Correspondence with `.uiflutmax` Keys

Since wire names are deterministically generated based on object names, decompiling the same `.maxpat` always produces the same wire names. This stabilizes `.uiflutmax` keys and enables Git tracking.

## Parser Architecture (E43)

### v0.4.0 Change: Pure Rust Recursive Descent Parser

The Tree-sitter (C generated code) based parser used through v0.3.0 has been replaced with a **Pure Rust recursive descent parser**.

```
v0.3.0: .flutmax -> [tree-sitter C parser] -> CST -> [parse.rs Rust] -> AST
v0.4.0: .flutmax -> [FlutmaxLexer Rust] -> Token[] -> [FlutmaxParser Rust] -> AST
```

### Motivation for Migration

1. **Elimination of C code dependency**: No longer requires building the `parser.c` generated by tree-sitter
2. **Improved error message accuracy**: Can report precise line/column numbers and error details
3. **Flexible token handling**: Naturally handles operator names, scientific notation, contextual keywords, etc.
4. **Multi-language binding support**: Pure Rust facilitates use via PyO3 (Python), wasm-bindgen (WASM), cbindgen (C FFI)

### Structure

```
crates/flutmax-parser/src/
├── lib.rs           # pub fn parse() entry point
├── tokens.rs        # TokenType enum, Token struct
├── lexer.rs         # FlutmaxLexer (tokenization)
├── parser.rs        # FlutmaxParser (recursive descent parser)
└── parse.rs         # Legacy tree-sitter version (for test comparison)
```

### Role of Tree-sitter

Tree-sitter is maintained exclusively for syntax highlighting in the VS Code extension. It has been removed from the compiler pipeline, and C compilation is no longer required at build time.

## Complete EBNF

```ebnf
(* === Top Level === *)
source_file          = { statement } ;

statement            = port_declaration
                     | wire_declaration
                     | destructuring_wire
                     | out_assignment
                     | direct_connection
                     | feedback_declaration
                     | feedback_assignment
                     | state_declaration
                     | state_assignment
                     | msg_declaration ;

(* === Port Declaration === *)
port_declaration     = direction ( explicit_port | implicit_port ) ;  (* v0.4.0: two forms *)
explicit_port        = INTEGER "(" IDENTIFIER ")" ":" type_name ";" ;
implicit_port        = IDENTIFIER ":" type_name ";" ;                 (* v0.4.0: implicit index *)
direction            = "in" | "out" ;
type_name            = "signal" | "float" | "int" | "bang" | "list" | "symbol" ;

(* === Wire Declaration === *)
wire_declaration     = "wire" IDENTIFIER "=" expression [ ".attr(" attr_list ")" ] ";" ;
attr_list            = attr_pair { "," attr_pair } ;
attr_pair            = IDENTIFIER ":" attr_value ;
attr_value           = number | string | IDENTIFIER ;
destructuring_wire   = "wire" "(" identifier_list ")" "=" expression ";" ;
identifier_list      = IDENTIFIER "," IDENTIFIER { "," IDENTIFIER } ;

(* === Output Assignment === *)
out_assignment       = "out" "[" INTEGER "]" "=" expression ";" ;

(* === Direct Connection (lvalue) === *)
direct_connection    = input_port_access "=" expression ";" ;
input_port_access    = IDENTIFIER ".in[" INTEGER "]" ;

(* === Feedback === *)
feedback_declaration = "feedback" IDENTIFIER ":" type_name ";" ;
feedback_assignment  = "feedback" IDENTIFIER "=" expression ";" ;

(* === State (Control-rate local state) === *)
state_declaration    = "state" IDENTIFIER ":" control_type "=" expression ";" ;
state_assignment     = "state" IDENTIFIER "=" expression ";" ;
control_type         = "float" | "int" | "bang" | "list" | "symbol" ;

(* === Message Box === *)
msg_declaration      = "msg" IDENTIFIER "=" STRING [ ".attr(" attr_list ")" ] ";" ;

(* === Expression === *)
expression           = object_call
                     | output_port_access
                     | tuple_expression
                     | name_or_ref       (* identifier / keyword-as-identifier *)
                     | number
                     | string ;

output_port_access   = object_name ".out[" INTEGER "]" ;

object_call          = object_name "(" [ argument_list ] ")" ;
object_name          = ( IDENTIFIER | OPERATOR | KEYWORD )   (* v0.4.0: operator names, contextual keywords *)
                       { "." ( IDENTIFIER | OPERATOR | KEYWORD | NUMBER_IDENT ) }
                       [ "=" ]                               (* trailing = : gbr.wind= etc. *)
                       [ "~" ] ;                             (* tilde: no whitespace allowed *)
argument_list        = expression { "," expression } ;
tuple_expression     = "(" expression "," expression { "," expression } ")" ;

name_or_ref          = IDENTIFIER | KEYWORD ;  (* keywords in expressions are treated as identifiers *)

(* === Tokens === *)
IDENTIFIER           = SIMPLE_IDENT ;                               (* Lexer outputs single segments *)
SIMPLE_IDENT         = LETTER { LETTER | DIGIT | "_" | "-" } ;     (* ADR-003: hyphens allowed *)
OPERATOR             = OPERATOR_CHAR { OPERATOR_CHAR | "=" } ;     (* v0.4.0: operator names *)
KEYWORD              = "in" | "out" | "wire" | "msg" | "feedback"  (* v0.4.0: contextual keywords *)
                     | "state" | "signal" | "float" | "int"
                     | "bang" | "list" | "symbol" ;
number               = [ "-" ] DIGIT { DIGIT }                     (* integer part *)
                       [ "." { DIGIT } ]                            (* fractional part (v0.4.0: trailing dot allowed) *)
                       [ ( "e" | "E" ) [ "+" | "-" ] DIGIT { DIGIT } ] ;  (* v0.4.0: scientific notation *)
string               = '"' { CHAR | ESCAPE } '"' ;
comment              = "//" { ANY except NEWLINE } ;                (* Lexer auto-skips *)

LETTER               = "a"-"z" | "A"-"Z" | "_" ;
DIGIT                = "0"-"9" ;
OPERATOR_CHAR        = "*" | "+" | "/" | "%" | "!" | "<" | ">" | "=" | "&" | "|" | "^" | "?" ;
ESCAPE               = "\\" ANY ;

(* === Notes === *)
(* - Lexer outputs dot `.` as an independent token; Parser assembles dot-delimited names *)
(* - Tilde `~` is also an independent token; Parser checks adjacency to combine with the name *)
(* - `.attr(` is recognized as a single DotAttrLParen token *)
(* - Keywords function as keywords only at statement-initial positions; in expressions they are treated as identifiers *)
```
